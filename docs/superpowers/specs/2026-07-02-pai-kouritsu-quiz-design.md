# 牌効率クイズ（何切る）設計書

## 背景・目的

麻雀の実力向上を目的としたWebブラウザアプリを新規作成する。第一弾として、牌効率（何切るクイズ）のトレーニング機能を作る。ランダムに生成された14枚の手牌から「シャンテン数を進め、かつ受け入れ枚数（ウケイレ）が最大になる打牌」を選ぶクイズを繰り返し解くことで、牌効率の感覚を鍛える。

既存の `麻雀`（大会スケジュール・点数管理）、`mahjong-table`（同種・Firebase連携版）はどちらも大会運営ツールであり、個人の牌効率トレーニングを目的としたものではないため、`麻雀強化` フォルダに独立した新規アプリとして作成する。

## 全体構成

`mahjong-table` プロジェクトと同様に、ビルドツールなしのファイル分割構成とする。

| ファイル | 役割 |
|---|---|
| `index.html` | HTML骨格。牌表示・成績表示・難易度選択などの空コンテナを配置し、CSS/JSを読み込む |
| `app.js` | シャンテン計算エンジン、問題生成、クイズ進行ロジック、描画、成績保存を含む全ロジック |
| `styles.css` | 牌カードのスタイル、スート別配色、レイアウト |

外部ライブラリ・Firebase・ビルドステップは使用しない。牌の見た目はUnicode麻雀絵文字（🀇🀈🀙…U+1F007〜）をCSSで装飾して表示する（画像素材なし）。

## データ表現

手牌は34種類の牌種に対する枚数配列 `counts[34]` で表現する。

- 0-8: 萬子1〜9
- 9-17: 筒子1〜9
- 18-26: 索子1〜9
- 27-33: 字牌（東南西北白發中）

## シャンテン数計算エンジン

`shanten(counts)` は以下3種類のシャンテン数の最小値を返す。この関数は13枚・14枚どちらの手牌にも同じロジックで適用できる（14枚で完成形に到達できればシャンテン数 -1 = 和了）。

- **通常形シャンテン**：面子・搭子（塔子）・雀頭への分解を再帰的に探索する古典的アルゴリズム。スートごとに可能な分解パターンを列挙し、雀頭候補を含めて組み合わせを比較して最小値を求める。
- **七対子シャンテン**：`6 - 対子数 + max(0, 7 - 保有種類数)`（対子数は最大7でカウント）
- **国士無双シャンテン**：`13 - 幺九牌の保有種類数 - (幺九牌に対子があれば1、なければ0)`

### 受け入れ枚数（ウケイレ）判定

13枚の手牌 `hand13` に対し、牌種 `t`（1〜34）を1枚加えたとき `shanten(hand13 + t) < shanten(hand13)` となる場合、`t` を「受け入れ牌」とする。受け入れ枚数は、各受け入れ牌について `4 - hand13内の保有枚数` を合計した値（山や他家の手牌は考慮しない、独立問題としての簡易計算）。

### 何切る判定

14枚の手牌に対し、13種類ありうる打牌候補それぞれについて：

1. 打牌後の13枚のシャンテン数を計算
2. その中で最小のシャンテン数を達成するグループに絞り込む
3. グループ内で受け入れ枚数が最大の打牌を「正解」とする（同点なら複数正解）

## 問題生成

1. 34種×4枚＝136枚の牌山からランダムに14枚を重複なく抽選する。
2. 難易度フィルタに応じて、生成した14枚の（打牌前の）シャンテン数が条件を満たすまで再抽選する。
   - 「0〜1シャンテン」：聴牌前後の実戦的な問題
   - 「2〜3シャンテン」：中盤の構想力を鍛える問題
   - 「絞り込みなし」：完全ランダム

## クイズ進行・UI

1. 生成された14枚の牌をUnicode牌カードとしてクリック可能な形で描画（スート別に配色）。
2. ユーザーが1枚クリックして打牌を選択する。
3. 即座に正誤を色で表示（正解グループ＝緑、それ以外＝赤）。
4. 解説エリアに、全13種の打牌候補を「シャンテン数・受け入れ枚数」降順でテーブル表示する。
5. 「次の問題へ」ボタンで新しい問題を生成する。
6. 画面上部に難易度フィルタのボタン（3段階）を常設する。

## 成績・永続化

`localStorage` に集計値を保存する（キー例: `mahjong_kiru_stats`）。

```
{
  attempts: number,       // 累計出題数
  correct: number,        // 累計正解数
  currentStreak: number,  // 現在の連続正解数
  bestStreak: number      // 自己ベスト連続正解数
}
```

難易度別の内訳は保存しない（全難易度合算のみ）。ページ上部に正答率・現在の連続正解数・自己ベストを常時表示し、ページ読み込み時に復元する。

## テスト・検証方針

自動テストフレームワークは導入しない（既存姉妹プロジェクトと同方針）。実装後、以下を手動検証する。

- 既知の答えが明らかな手牌（例：国士無双13面待ち、一般的な聴牌形）でシャンテン数・受け入れ枚数が正しく算出されることをブラウザのコンソール等で確認する。
- クイズを複数回実施し、正誤判定・解説表示・成績更新・難易度フィルタが期待通り動作することをブラウザで確認する。

## スコープ外（今回やらないこと）

- CPU対戦シミュレーター、点数計算学習、牌譜分析など他のトレーニングモードは別プロジェクト・別スペックとして扱う。
- 赤ドラ・ドラ表示、点数期待値を考慮した「実戦的な正解判定」は行わない（受け入れ枚数最大化のみ）。
- 難易度別の成績内訳保存は行わない。
